# Admin Output Voice

这个 Skill 管的是「AI 看完规范之后，给用户回的文字长什么样」。它不替换任何规则，只管产出的语气。

## 核心判断：规范是给 AI 看的，回答是给人看的

这套后台设计 Skills 里的文字服务两类读者：

- **规范本身**（`SKILL.md`、`references/*.md`、`docs/*.md`）是给 AI 看的。它用「必须 / 应 / 禁止」和「S0 / P1 / PT1 / CC1」这种高密度的规范书腔调，是为了让 AI 不放飞——这种腔调**必要**，不要去改。
- **AI 对用户的回答** 是给人看的。人读「违反 XX 承诺 / 业务正确性级问题 / 结构层 + 契约层 + 数据正确性层都存在阻断问题」会头疼，读完还要在脑子里翻译成「哦，意思是有 bug」。

所以 AI 把规范翻译成回答时，要做一层转译：把内部代号、规范词、并列名词堆，换成人能一眼看懂的句子。**转译不做，就是不说人话。**

## 何时使用

- 所有 admin 设计 skill 给用户的可见输出——尤其是 `admin-design-review` 和 `admin-design-orchestrator`，它们产出的文字量最大，也最容易翻车
- 澄清提问、组件推荐、分流决策、修复建议、残余风险——任何最终落在对话里的段落

内部思考、内部 checklist、规范文件本身不受此 Skill 约束。

## 七条产出规则

### 1. 结论先行

第一句话就是结论或最重要的动作。不做「审查对象 / 技术栈 / 业务定位」的登记开场。

- 反例：前七行登记背景信息，读者读到第 8 行才看到「结论」标签
- 正例：「这页有一个必须第一时间修的 bug——渠道筛选器的 ID 冲突会让用户选错一批账单。除了它，页面整体结构也错位了……」

如果必须给背景，放在结论之后一段里，不放在开头。

### 2. 代号只留严重度

给用户的输出里，**只保留 S0 / S1 / S2 / S3** 这四个严重度标签——它们直接表达优先级，读者不用查表。

其他内部代号（`P1` / `P2` / `PT1` / `CC1` / `ST1` / `VS1` 等）**一律不出现**。需要跨段落引用前面提过的问题时，用小节标题或描述，不用编号：

- 反例：「修 CC4 渠道筛选 ID 冲突，再修 PT2 结果摘要」
- 正例：「先修那个渠道筛选 bug（上面标了 S0 那条），然后在表格上方加一排汇总卡」

内部分层术语（Principles / Patterns / Contracts / Style / Visualization / Motion）做单独裸词还可以接受，但**不连成**「结构层 + 契约层 + 数据正确性层」这种并列堆叠。

### 3. What 必带 Why 和用户影响

每条问题除了讲「代码或设计是什么样」，必须再带一句「会对用户 / 业务造成什么」。

| 只讲 what（不够）               | 补上 why 和影响（够）                                        |
| -------------------------------- | ------------------------------------------------------------ |
| 28 列平铺，无汇总                 | 财务打开要拖横条找关键列，汇总还要导 Excel 自己算        |
| 每行跑一次 `SysUser::query()`     | 这是 N+1——页面越用越慢，行数一上来直接拖垮数据库             |
| `computeReceivableBillUnclearedAmount` 挂在 Controller 上 | 领域公式被写进了控制器，哪天改公式要翻遍所有控制器，容易漏改 |

不给原因的问题条目是"规则打卡"，不是"帮人做事"。

### 4. 避免规范书腔调

下列词语进黑名单，**不在给用户的回答里用**（对 AI 内部规则文件仍可用）：

- **法律条文腔**：违反、偏离、不符、背离、承诺（动名词）、履行
- **官腔动词**：兜底、硬塞、承载、埋了、下沉、上抛、收口（作为纯书面隐喻时）
- **造词后缀**：X 级问题、X 承诺、X 硬伤（"数据正确性级问题 / 5 秒承诺 / 业务正确性硬伤" 这类）

换成**发生了什么 + 会出什么事**的大白话：

| 规范腔                     | 人话                                             |
| -------------------------- | ------------------------------------------------ |
| 违反 5 秒承诺               | 打开 5 秒内看不出重点                             |
| 业务正确性级问题            | 会让财务拿错数据去对账的硬 bug                    |
| 口径硬塞展示层              | 领域计算被写进了控制器里                          |
| 埋了口径分裂风险            | 数据对不上的时候谁也说不清按哪个公式算            |
| 偏离了核销跟踪的业务任务    | 业务上要做的是跟核销，现在做成了字段目录         |

完整词表和替换看 `references/banned-words.md`。

### 5. 不堆加号并列

并列名词最多 3 个，用顿号或"和"；4 个以上拆成句子。

- 反例：「场景堆砌 + 正确性漏洞 + N+1 + 口径硬塞展示层」
- 正例：「组件契约上有四处问题：渠道筛选会选错、每行重复查询用户、领域公式写在控制器里、命名跟文档对不上」

英文模板名也一样：

- 反例：「Filter + Summary + Status Tabs + Table」
- 正例：「筛选 + 汇总卡 + 状态 Tab + 表格（英文叫 Filter + Summary + Status Tabs + Table）」

一段话里不要出现超过两个 `+` 号。

### 6. 术语译名统一，首次出现附解释

所有抽象模式名**第一次出现**时，带一次中文译名或一句解释，之后就只用一套叫法：

| 英文               | 中文        | 之后用哪个 |
| ------------------ | ----------- | ---------- |
| `Queue Block`      | 队列分组卡   | 中文       |
| `Signal Panel`     | 信号总览     | 中文       |
| `Scan Table`       | 扫读列表     | 中文       |
| `Status Rail`      | 进度轨       | 中文       |
| `Pill`             | 徽章         | 中文       |
| `KPI 卡`           | 汇总指标卡   | 中文       |
| `Detail Drawer`    | 详情抽屉     | 中文       |

完整对照看 `references/glossary.md`。

不要一会儿 `Pill` 一会儿"徽章"一会儿"标签"——**同一份回答里一个概念只用一套叫法**。

代码字段名（`settlement_status` / `bill_type`）保持原文不翻译，但周围要有中文上下文说它是什么。

### 7. Bullet 有节制

- **单块 bullet 最多 5 条**。超过就分组、改成表格，或合并成段落
- **单份回答大表格最多 1 个**。超过要拆章节
- **bullet 块之间要有叙述段落承接**，不要连续三个 bullet 块硬接
- **评审 / 诊断类回答**：开头一段人话交代结论 + 中间按严重度组织问题 + 结尾一段人话收束。不要起手就 bullet、结束就 bullet

判断标准：把回答打印出来，第一眼看到的是密集项目符号还是文字段落？如果是前者，拆。

## 硬规则

- 不把 `admin-design-review` 或 `admin-design-orchestrator` 的 `输出格式` 字段列表当作最终回答直接交给用户——那是内部结构，要填好再翻译成段落
- 不把 review 写成只有表格没有叙述
- 不在给用户的输出里保留 `P1 / PT1 / CC1 / ST1 / VS1` 这类内部编号
- 不用黑名单词表里的词（规范书腔调）
- 每个问题条目至少要有一句"对用户或业务的影响"
- 不在一份回答里出现超过 `1` 个大表格或连续 `3` 块无叙述承接的 bullet

## 使用提醒

- 本 Skill 适用于所有对用户的可见产出，不适用于 AI 内部思考、规范文件本身或硬规则的措辞
- 黑名单词表和替换词看 [references/banned-words.md](references/banned-words.md)
- 术语中英文对照看 [references/glossary.md](references/glossary.md)
- 产出前后对照示例看 [references/before-after-examples.md](references/before-after-examples.md)，包含 review 报告、分流决策、组件推荐三种典型场景
- 这个 SKILL.md 本身就按 7 条规则写作，可以直接当文体示例

## 默认做法

AI 写完一段给用户的回答之后、发送之前，自己读一遍，过六问自检：

1. 第一句有没有直接给结论？
2. 有没有保留不该暴露的内部代号（`P1 / PT1 / CC1` 等）？
3. 每条问题有没有带一句"对用户的影响"？
4. 有没有规范书腔调词（违反 / 兜底 / X 级问题等）？
5. 同一概念有没有混用多种叫法？
6. bullet 块之间有没有叙述承接，还是连着三块硬怼？

任何一项不过，改完再发。
