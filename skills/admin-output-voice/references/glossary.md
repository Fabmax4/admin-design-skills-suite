# 术语中英文对照

这是整套 admin 设计 Skills 对外输出时的**术语统一表**。AI 给用户写回答时，按这张表用词——首次出现用中文，后续只用中文；英文括注只在首次出现时出现一次。

## 严重度（review 输出）

| 代号  | 含义                                 | 要不要出现在给用户的回答里                |
| ----- | ------------------------------------ | ----------------------------------------- |
| `S0`  | 阻断 / 必须先修                       | **是**，推荐保留。直接表达优先级           |
| `S1`  | 高优                                  | **是**                                     |
| `S2`  | 中优                                  | **是**                                     |
| `S3`  | 低优                                  | **是**                                     |
| `P1`-`P3`  | 原则层问题编号                    | **否**                                     |
| `PT1`-`PT3`| 模式层问题编号                    | **否**                                     |
| `CC1`-`CC6`| 契约层问题编号                    | **否**                                     |
| `ST1`-`ST4`| 样式层问题编号                    | **否**                                     |
| `VS1`-`VS3`| 可视化层问题编号                  | **否**                                     |

需要跨段落引用一条具体问题时，用描述而不是代号：

- ❌「修 CC4」 ✅「先修那个渠道筛选 bug」
- ❌「参考 PT2 增加结果摘要」 ✅「在表格上方加一排汇总卡（上面列的第二条问题）」

## 分层名词

这套 skill 的六层（原则、模式、契约、样式、可视化、动效）在给用户的回答里：

- 单独出现可以用裸词：「这是**原则**层面的问题」
- **不要连成并列**：「结构层 + 契约层 + 数据正确性层 + 原则层」这种四五个"X 层"堆在一起，绝对不要

两个层并列可以：「原则和模式上都有问题」。三个或更多就拆成句子。

## 维护规则

- 新增的抽象模式名进这张表之前，先确定中文译名
- 同一个中文译名不要对应两个英文——有歧义就换译名
- skill 的 `SKILL.md` / `references/*.md` 可以继续用英文原名做内部索引；这张表只约束**对用户的输出**
