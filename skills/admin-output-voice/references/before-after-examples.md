# 产出前后对照示例

这份文件收本 Skill 上线时的典型"之前 vs 之后"对照，供 AI 在写长回答前快速看一眼参考。三个典型场景：review 报告、分流决策、组件推荐。

---

## 示例 1：review 报告开头

### 之前（`admin-design-review` 生成的规范腔版本）

```
审查对象
- 页面类型：账单列表页
- 技术栈：Vue 3 + ant-design-vue
- 业务定位：应收账单管理与核销

前置澄清是否充分
- 部分充分。核销跟踪任务未在输入中显式确认。

结论
- 本次审查识别出 P1 / P2 / PT1 / CC1 / CC4 / VS1 六项关键问题。
- 结构层、契约层、数据正确性层均存在阻断问题。
- 违反 5 秒承诺与数据一致性承诺。

主要问题
- P1：页面主任务未明。
- CC4：渠道筛选器 ID 冲突。
- PT1：场景堆砌 + 正确性漏洞 + N+1 + 口径硬塞展示层。
- VS1：28 列平铺，无汇总。
...
```

### 之后（套用 `admin-output-voice`）

```
这页有一个必须第一时间修的硬 bug：渠道筛选器传给后端的 ID 会撞另一批单据的 ID，
用户选"线上渠道"可能会把线下账单也选进来。做财务对账的时候这种错会直接让
报表对不上，一定先修它。

除了这个 bug，页面还有三件事不对：

- 最该做的事在页面上看不出来——这页的业务目标是"跟核销进度"，但现在打开
  来就是 28 列平铺的字段目录，财务得拖横条找关键列，没地方一眼看出"今天
  有多少单没核销完"
- 每打开一次列表都会给每一行跑一次查用户的查询（这是 N+1），行数一上来
  页面会明显变卡
- 金额相关的计算公式写在 Controller 里，将来口径要改得翻一遍控制器，很容易漏改

按严重度分：
- S0：渠道筛选 ID 冲突、金额公式写在 Controller（数据会对不上）
- S1：页面主任务不明、28 列无汇总、N+1 查询
- S2：列宽和列头样式不一致
```

### 差别在哪

- **第一句直接给结论**（先说最要紧的 bug），不是"审查对象"登记
- **代号消失**，只留 S0 / S1 / S2 做优先级标签
- **每条问题都带"对用户/业务的影响"**：选错账单 → 报表对不上，28 列平铺 → 拖横条找不到重点
- **规范腔被换掉**："违反 5 秒承诺" → "没地方一眼看出..."；"口径硬塞展示层" → "金额公式写在 Controller 里"
- **开头一段叙述承接**，然后才是 bullet；不是起手就 bullet

## 示例 2：分流决策（`admin-design-orchestrator`）

### 之前

```
任务阶段：结构设计
输入是否充分：不充分
待澄清问题：
- 页面范式未明
- 主任务未明
- 使用角色未明
建议调用的 Skill：
- design-principles
- admin-design-patterns
调用顺序：design-principles → admin-design-patterns
本轮重点：方向判断
暂不进入的层：style-guardrails / admin-visualization / admin-motion / admin-component-contracts
下一步：补齐输入后再分流
```

### 之后

```
先不动手画页面——你给的信息还差一些关键问题。我需要先问清楚三件事才能
选对页面范式：

1. 这页打开第一件事是什么？看数 / 处理待办 / 做配置？
2. 主要谁用？财务、运营、管理员，角色不同页面重点差很多
3. 有没有已经在跑的版本？有的话我直接按它改，没有就从空白开始

这三问回答完，我先进「设计原则」和「设计模式」这两个 skill 定大方向，
样式、组件、可视化、动效都先搁置——结构没定之前聊样式容易白做。
```

### 差别在哪

- 把 8 条分项列表**翻译成一段话 + 三个清晰的问题**
- **解释了为什么暂不进入其他层**（「结构没定之前聊样式容易白做」），而不是只甩"暂不进入的层：..."
- **避免了"任务阶段 / 输入是否充分 / 待澄清问题"这种规范字段名**，把它们都溶进自然语句

## 示例 3：组件推荐（`admin-component-contracts`）

### 之前

```
推荐契约：Detail Drawer + Bulk Action Bar + Filter Panel
对应 ant-design-vue 实现：a-drawer + 自定义 ActionBar + a-form 作为筛选容器
契约项：
- 输入契约：list data, filter schema, selected rows
- 事件契约：on-select, on-filter-change, on-bulk-action
浮层归属：detail 走 Drawer，bulk confirm 走 Popconfirm
可访问性底线：键盘可触达、焦点回到触发元素
```

### 之后

```
这页的核心动作是"批量核销"，推荐三件套：

- **详情抽屉**：用户点一行就从右侧滑出详情（用 `a-drawer`）。比模态框好在不遮
  挡列表，可以边看详情边切换其他行
- **批量操作栏**：选了行之后在底部出现一条条浮出来的操作条，集中放"批量核销"
  "批量导出"这几个动作（自己搭一个 ActionBar 组件，ant-design-vue 没现成的）
- **筛选面板**：表格上方一行筛选器，用 `a-form` 的 inline 模式装就行

需要注意两件小事：
1. 批量操作如果是"删除"或不可逆动作，在触发时用 `a-popconfirm` 再确认一次
2. 抽屉关闭后焦点要回到那一行，不然键盘用户会丢失位置
```

### 差别在哪

- 英文契约名**首次出现配中文和一句用法解释**，之后就用中文
- 把"输入契约 / 事件契约 / 浮层归属 / 可访问性底线"这种**结构字段**溶进了自然段落
- **讲清楚为什么这样选**："比模态框好在不遮挡列表"「不然键盘用户会丢失位置」
- **ant-design-vue 的具体组件**（`a-drawer / a-form / a-popconfirm`）作为实现起点给出来，但不是强迫症式对齐契约表

## 通用差别提炼

这三个例子有共同的转换路径，供 AI 写回答时快速自检：

1. **把"字段列表"改成"话题段落"**——`审查对象 / 技术栈 / 业务定位 / 结论 / 主要问题 / ...` 是内部结构，不是最终回答
2. **先讲一句最要紧的**——结论 / 最紧急 bug / 先做什么，永远在开头
3. **每件事配一句"影响"或"原因"**——不是"做错了"而是"做错会 XX"
4. **代号转描述**——`CC4` 变"那个渠道筛选 bug"，`S0` 保留
5. **规范词转日常词**——"违反 XX 承诺" 变 "没做到 XX"，"硬塞" 变 "写进了"
6. **英文首次附中文，之后只用中文**，代码字段名保持原文

## 写长回答前的五秒钟自检

写完一段准备发出去之前，扫一眼这份文件最上面的"之前"版本长什么样。如果你的回答读起来跟"之前"版本像，停下来重写成"之后"风格。
