# 黑名单词表

这些词在给用户的回答里**不要用**。写成给 AI 看的规则文件时可以用，写成给人看的回答时必须换掉。

分类组织，每一条给替换思路。

## 法律条文腔

规范书、政府公文的常用词。用在 AI 产出的回答里会让读者感觉隔着一层玻璃。

| 黑名单              | 换成                                     |
| ------------------ | ---------------------------------------- |
| 违反 XX 规范        | 跟 XX 对不上 / 没按 XX 做                 |
| 违反 XX 承诺        | XX 目标没做到 / 达不到 XX 要求            |
| 偏离了 XX           | 做成 XX 了 / 跑偏到 XX 了                 |
| 不符 XX              | 跟 XX 不一致 / 没满足 XX                 |
| 背离                | 反过来了 / 走到反方向了                   |
| 承诺（动名词）       | 目标 / 要做到的事                         |
| 履行                | 按 ... 做 / 实现                         |

## 官腔动词

这些词作为比喻用在书面规范里还好，作为动词出现在对人回答的句子里会显得很"PPT 腔"。

| 黑名单              | 换成                                     |
| ------------------ | ---------------------------------------- |
| 兜底                | 保证 / 垫底的做法 / 最后一道防线         |
| 硬塞                | 写进了 / 放进了                           |
| 承载                | 装 / 包含 / 负责表达                     |
| 埋了 XX 风险         | 会出 XX 问题 / 以后可能踩 XX 坑           |
| 下沉                | 放到下层 / 移到 XX                        |
| 上抛                | 往上抛 / 往上传                           |
| 收口                | 收尾 / 收到一起                           |
| 沉淀                | 攒下来 / 存下来                           |
| 打通                | 接上 / 连起来                             |
| 前置                | 提前 / 先做                               |
| 后置                | 往后挪 / 延后                             |

注：「兜底 / 下沉」这类词在工程讨论中有时是术语而非官腔，当且仅当表达"一种具体的技术手段"时可以保留，**但要带一句解释这个手段具体是什么**。

## 造词后缀

用抽象后缀把名词造成"看起来很专业"的短语。AI 最容易学会这种造词法。

| 黑名单              | 换成                                     |
| ------------------ | ---------------------------------------- |
| XX 级问题            | XX 的问题 / 会 XX 的 bug                  |
| XX 承诺              | XX 目标 / 要做到 XX                       |
| XX 硬伤              | 一个硬 bug / 过不去的坎                   |
| XX 漏洞（非安全语境） | XX 的问题 / 有个 bug                      |
| XX 化（泛用）        | 变成 XX / 让 XX                          |

具体例子：

- ❌「业务正确性级问题」 ✅「会让业务数据出错的 bug」
- ❌「5 秒承诺」 ✅「5 秒内要能看到重点」
- ❌「数据一致性硬伤」 ✅「数据对不上的坑」
- ❌「结构正确性漏洞」 ✅「结构上的问题」

## 名词堆叠

把内部分层名词用 `+` 号或顿号并列，读者要一个个翻译。

| 黑名单                                       | 换成                                        |
| -------------------------------------------- | ------------------------------------------- |
| 结构层 + 契约层 + 数据正确性层都有问题          | 结构、组件契约、数据正确性上都有问题         |
| 原则层和模式层不先修                           | 原则和模式没修之前                           |
| 场景堆砌 + 正确性漏洞 + N+1 + 口径硬塞展示层    | 契约上有四处问题（后面列出来）               |

一句话里不要出现超过两个 `+` 号。超过就换成句子或列表。

## 中英混搭

| 黑名单                       | 换成                                     |
| ---------------------------- | ---------------------------------------- |
| 使用 Pill                     | 用徽章（Pill）                            |
| 放 KPI 区                     | 放一排汇总指标卡                          |
| 套 Filter + Table 范式         | 套"筛选 + 表格"这套范式                  |
| 用 Drawer 代替 Modal           | 用抽屉代替模态框                          |

规则：**第一次出现**时中文为主、英文附注一次；之后统一用中文；代码字段名（`settlement_status` 这种）保持原文。完整译名对照看 [glossary.md](glossary.md)。

## 被动语态滥用

被动句在公文里显得"客观"，但读者需要主语来理解"谁做了什么"。

| 黑名单                               | 换成                                     |
| ------------------------------------ | ---------------------------------------- |
| XX 从未在页面层被回答                  | 页面上没有地方告诉用户 XX                  |
| 核销跟踪未在页面中被表达               | 核销跟踪这件事在页面上看不出来             |
| 该假设需要在修复前被用户确认           | 动手之前得先跟用户确认这个假设             |

规则：每三句话里不要有超过一个被动句。

## 自检触发

如果你写的一段话里出现了上面任何一个词，停一下，换成替换词再继续。如果某个词在具体上下文里确实不好换（比如工程术语），**在同一段里必须附一句自然语言解释它是什么**。
