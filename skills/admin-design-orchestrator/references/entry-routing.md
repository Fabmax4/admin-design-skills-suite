# 入口分流规则

## 文件定位

这是规则层文件，用来定义总入口 Skill 的分流逻辑。

- 这里应保持行业中立。
- 具体行业语境和示例请求留在案例层文件中，例如 `example-call-set.md` 和 `routing-sample-set.yaml`。

## 混合意图总规则

如果用户请求里同时出现两个及以上目标，优先判断：

1. 哪个目标被 `先`、`先别`、`先不要`、`优先`、`先帮我` 明确标成主任务
2. 哪个目标会决定后续所有层是否成立
3. 哪个目标属于更上层

默认规则：

- 主意图先路由
- 次意图延后到 `下一步`
- 只有主意图已经稳定，才进入次意图对应层

不要因为用户提了两个目标，就把两个链路同时完整跑完。

## 冲突意图总规则

如果用户请求里同时出现互相冲突的要求，优先判断：

1. 哪个要求是硬约束
2. 哪个要求只是增强诉求
3. 两者是否能在不违反硬约束的前提下同时成立

默认规则：

- `不要`、`先别`、`不重做`、`保持不变` 这类约束优先级更高
- 如果冲突发生在同一层，先在该层收边界
- 如果冲突横跨多层，优先进入能暴露冲突的更高层或 review
- 如果两者无法同时成立，不假装都能完成，而是先输出冲突判断和下一步

## 输入信息不足

特征：

- 用户只给业务目标
- 没说清页面类型
- 没说清第一优先动作
- 没说清使用角色或成功标准

起始 Skill：

- 暂不直接进入分流

规则：

- 先请求澄清
- 如果用户在澄清后仍无法准确描述，再进入 `admin-design-principles`
- 只有当页面范式、主任务或使用角色至少清楚到能判断起始层时，才继续分流

## 明确页面，但结构未稳

特征：

- 用户已经说了是工作台、列表、详情、编辑或某类后台页
- 但模式块和区块职责还没稳定

起始 Skill：

- `admin-design-patterns`
- `admin-design-components`

## 结构稳定，开始收视觉

特征：

- 页面骨架已经明确
- 现在要控制色彩、圆角、密度、版式

起始 Skill：

- `admin-design-style`

## 数据密集，开始决定表达方式

特征：

- 页面需要比较、趋势、分布或阈值判断
- 需要决定是否用图表

起始 Skill：

- `admin-design-visualization`

## 页面已经成型，开始补反馈

特征：

- 页面结构稳定
- 需要决定是否要转场、状态反馈、焦点变化

起始 Skill：

- `admin-design-motion`

## 需要验收或 review

特征：

- 用户明确说要 review、验收、把关
- 或方案已经足够完整

起始 Skill：

- `admin-design-review`

## mixed review

特征：

- 用户一边要求 review
- 一边要求你给下一轮推进链路、下一版结构方向或后续 Skill 顺序

起始 Skill：

- `admin-design-review`

规则：

- 先 review
- 再在 review 结论里给出下一轮 Skill 链路
- 不要因为“还要给下一步”就重新跑一遍完整设计流程

## 冲突中的冻结约束

特征：

- 用户明确说结构、视觉、动效或功能先不要动
- 同时又提出会打破该边界的优化请求

起始 Skill：

- 同层冲突：进入该层 Skill
- 跨层冲突：进入 `admin-design-review`

规则：

- 先判断冻结约束是否允许继续优化
- 不允许时，先输出冲突和建议，不直接往下层走

## 组件 / 规范抽象

特征：

- 用户要抽象后台组件、模式或规范
- 目标不是单页，而是可复用结构

起始 Skill：

- `admin-design-patterns`
- `admin-design-components`

## Selection Action Bar / 选择操作栏

特征：

- 用户提到表格多选、已选预览、选择汇总、批量操作、底部操作栏、sticky 操作栏、开票、结算、导出、确认等选中后主动作
- 或用户要求把某个已选栏、批量栏、底栏沉淀为规范
- 也可识别 `Selection Summary Bar`、`Batch Action Bar`、`Bulk Action Bar`、`Selected Rows Toolbar` 等叫法

起始 Skill：

- `admin-design-components`
- `admin-design-style`
- `admin-design-review`

规则：

- 先定义 `Selection Action Bar` 的输入、状态、主动作和禁用原因
- 再收视觉表面，判断它是页面级操作 chrome 还是容器内工具条
- 最后用 review 检查是否遮挡表格、空态、分页和横向滚动控件

## 总入口拦截条件

出现以下任一情况时，总入口应先停在澄清，不继续分流：

- 无法判断页面服务的是谁
- 无法判断页面的第一优先动作
- 无法判断这是设计任务、评审任务还是抽象任务
- 无法判断成功标准或关键约束
