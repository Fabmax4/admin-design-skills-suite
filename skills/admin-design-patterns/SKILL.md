# Admin Design Patterns

先做结构判断，再做组件决定。这个 Skill 用来约束 AI 在管理后台设计中，先判断页面范式和模式块，而不是一上来发明业务组件。

## 何时使用

- 用户要新做一个后台页面
- 用户要重构后台信息架构或页面布局
- 用户给的是业务场景，但还没有稳定结构
- 需要判断一个“新组件”是否其实只是旧模式换了业务语义

## 工作流程

1. 先判断页面范式
   只在 `工作台 / 查询列表 / 详情页 / 编辑页 / 配置页 / 异常处理台 / 资源运营台 / 协同台` 里选最接近的一类。
2. 找出第一优先动作
   说明这个页面最重要的处理动作是什么；如果说不清，就不要继续下钻组件。
3. 先校正信息架构与寻址
   如果这页属于现有导航体系，先确认导航分组、进入路径、返回路径和详情进入方式是否与同类页面一致。
4. 选择模式块
   先选 `1` 个主模式块，再选 `0-2` 个辅助模式块。不要把所有模式都塞进一个页面。
5. 回收业务命名
   把“对象卡”“风险卡”“位置卡”这类业务叫法回收到已有模式名；如果只是文案变了，就不要新建模式。
6. 输出结构结论
   明确页面范式、主模式块、辅助模式块、主动作、次级信息区，以及是否真的存在模式缺口。

## 输出格式

- `页面范式`
- `第一优先动作`
- `主模式块`
- `辅助模式块`
- `业务语义包装`
- `不该新增的组件`
- `真的缺失的模式`

## 硬规则

- 不先按业务名命名组件
- 不把配置页做成详情页
- 不把编辑页做成线性审批流，除非用户明确要步骤编排
- 不为了“丰富”而堆更多卡片
- 如果只是业务语义变化，不新增模式
- 不为单页局部优化打破整体导航和信息架构的寻址惯性
- 不把行业专有名词直接写成模式层的通用术语

## 使用提醒

- 最短发起格式可写成 `/admin-design-patterns 收页面结构` 或 `/admin-design-patterns shape page structure`
- 页面结构不清时，先读 `references/page-paradigms.md`
- 模式块选择不清时，先读 `references/pattern-catalog.md`
- 判断是否真的要新建模式时，读 `references/review-checklist.md`

## 默认做法

如果用户只给一句话，且当前输入不足以判断页面结构，默认先回答这 5 个问题：

1. 这是哪种页面范式？
2. 这个页面的第一优先动作是什么？
3. 需要哪些模式块？
4. 这些模式块里哪些已经存在？
5. 新需求是真的缺模式，还是只缺业务文案？
