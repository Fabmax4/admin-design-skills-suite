# 修复顺序

## 默认顺序

1. 先修 `Principles`
2. 再修 `Patterns`
3. 再修 `Component Contracts`
4. 再修 `Style`
5. 再修 `Visualization`
6. 最后修 `Motion`

## 原因

- 原则不清，后面的所有优化都会偏题
- 模式错了，组件和样式都会建立在错误结构上
- 契约不清，复用性会持续失控
- 样式和表达应服务于结构，而不是反过来主导设计

## 例外情况

- 如果页面本身结构正确，只是视觉失控，可以从 `Style` 开始
- 如果页面已稳定上线，只存在反馈问题，可以直接修 `Motion`
- 但在给出例外前，必须明确说明为什么不用默认顺序
