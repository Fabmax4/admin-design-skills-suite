# 总审查清单

每条 check 后面标注的严重度是**建议起点**，对照 `severity-model.md` 的 S0-S3 定义：

- **S0**：阻断，页面核心任务无法完成，必须先修
- **S1**：高优，结构/模式/契约或可访问性底线有问题
- **S2**：中优，视觉、可视化、动效表达失衡
- **S3**：低优，细节一致性

实际评审时可根据影响范围上调或下调。

## 1. Principles

- 输入信息不足时是否先经过澄清或启发式提问？ — 未澄清直接设计 → **S0**
- 页面任务是否明确？ — 不明确 → **S0**
- 第一优先动作是否明确？ — 不明确 → **S0**
- 异常、边界和恢复链路是否被纳入？ — 缺失 → **S1**
- 信息层级是否清晰？ — 同权堆放 → **S1**

## 2. Patterns

- 页面范式判断是否正确？ — 错位 → **S1**
- 主模式块和辅助模式块是否清晰？ — 模糊 → **S1**
- 是否存在"换业务名但没换模式"的伪新组件？ — 存在 → **S1**
- 导航分组、进入路径、返回路径是否符合用户肌肉记忆？ — 不一致 → **S1**

## 3. Component Contracts

- 组件是否有清晰的最小输入、状态和动作？ — 无契约 → **S1**
- 组件命名是否来自模式而不是业务名？ — 业务名为主 → **S2**
- 是否出现只有名字没有契约的组件？ — 出现 → **S1**
- 表格是否优先保障连续阅读惯性，减少单元格随机性？ — 单元格逐字换行或随机样式 → **S2**
- 列数超过 `15` 时，是否支持个性化显示列和列排序？ — 不支持 → **S2**

## 4. Style

- 页面是否仍然保持后台骨架？ — 营销页化 → **S2**
- 色彩、圆角、密度是否受控？ — 失控 → **S2**
- 是否出现过度装饰、过大留白、过重阴影？ — 出现 → **S2**
- 提示、提醒、摘要是否优先写清数量、时限和范围？ — 纯定性 → **S2**
- 可访问性底线是否满足？ — 不满足 → **S1**（对比度、焦点、降级属于底线）
- 中文徽章/标签下方是否叠加了原始英文枚举值（如 `SUCCESS / DEDUCTED`）？ — 暴露冗余英文 → **S2**（属于 UI 文案层的规范腔，见 `admin-output-voice`）
- 状态徽章的内边距是否过大，撑起一个明显比文字大的盒子？ — 不跟 `ant-design-vue` `<Tag>` 默认档位 → **S3**
- 按钮、标签、空状态文字是否中英混搭（`Submit 提交` / `No data` 等）？ — 混搭（且非品牌名/代码字段名）→ **S3**
- 按钮、徽章、链接、状态文字是否在**默认态**达到 WCAG AA 对比度（正文 `4.5:1`，大字/UI 组件边界 `3:1`）？用 DevTools / axe 实测过一次 — 不达标 → **S1**（可访问性底线）
- hover / focus 是否承担了"从不可见到可见"的揭示功能（文字默认几乎同色、悬浮才显形）？ — 是 → **S1**（违反默认态可读性，键盘/触屏用户受伤）
- 表格主键（订单号、单据号、用户 ID）、金额、状态等关键信息是否被装饰色块、主题色块、打码块**整体覆盖**，要 hover 才能看到？ — 是 → **S0**（默认态信息丢失，阻断核心任务）

## 5. Visualization

- 图形是否真的支持比较、判断或下钻？ — 不支持 → **S2**
- 本应留在指标卡或表格的信息是否被硬做成图？ — 硬做 → **S2**
- 图表是否脱离动作入口？ — 脱离 → **S2**

## 6. Motion

- 动效是否只服务于状态变化？ — 存在感导向 → **S2**
- 是否存在装饰性动画、过长转场或统一漂浮？ — 存在 → **S2**
- 在降级条件下状态是否依然清晰？ — 降级失效 → **S1**（可访问性底线）

## 审查顺序

永远按 `Principles -> Patterns -> Component Contracts -> Style -> Visualization -> Motion` 的顺序思考。
不要先盯动效和细节，再回头补结构。

严重度排序同理：**先解决 S0，再修 S1，最后处理 S2/S3**，顺序不可颠倒。
