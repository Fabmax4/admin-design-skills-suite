# 总审查清单

## 1. Principles

- 输入信息不足时是否先经过澄清或启发式提问？
- 页面任务是否明确？
- 第一优先动作是否明确？
- 异常、边界和恢复链路是否被纳入？
- 信息层级是否清晰？

## 2. Patterns

- 页面范式判断是否正确？
- 主模式块和辅助模式块是否清晰？
- 是否存在“换业务名但没换模式”的伪新组件？
- 导航分组、进入路径、返回路径是否符合用户肌肉记忆？

## 3. Component Contracts

- 组件是否有清晰的最小输入、状态和动作？
- 组件命名是否来自模式而不是业务名？
- 是否出现只有名字没有契约的组件？
- 表格是否优先保障连续阅读惯性，减少单元格随机性？
- 列数超过 `15` 时，是否支持个性化显示列和列排序？
- 表格多选后的 `Selection Action Bar` 是否说明已选数量、作用范围、跨页选择语义、关键金额或影响指标、主动作和禁用原因？

## 4. Style

- 页面是否仍然保持后台骨架？
- 色彩、圆角、密度是否受控？
- 是否出现过度装饰、过大留白、过重阴影？
- 提示、提醒、摘要是否优先写清数量、时限和范围？
- 页面级 `Selection Action Bar` 是否作为操作 chrome 呈现，而不是普通卡片；是否避免遮挡表格、空态、分页和横向滚动控件？
- 可访问性底线是否满足？

## 5. Visualization

- 图形是否真的支持比较、判断或下钻？
- 本应留在指标卡或表格的信息是否被硬做成图？
- 图表是否脱离动作入口？

## 6. Motion

- 动效是否只服务于状态变化？
- 是否存在装饰性动画、过长转场或统一漂浮？
- 在降级条件下状态是否依然清晰？

## 审查顺序

永远按 `Principles -> Patterns -> Component Contracts -> Style -> Visualization -> Motion` 的顺序思考。
不要先盯动效和细节，再回头补结构。
