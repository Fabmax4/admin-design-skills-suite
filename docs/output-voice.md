# Output Voice

这些规则约束 AI 把后台设计规范翻译成给用户看的回答。规范文件可以高密度、强约束；最终回复必须自然、可执行、少术语。

## 产出规则

1. 结论先行。第一句话直接说最重要的判断或下一步动作，不用“审查对象 / 技术栈 / 任务阶段”登记开场。
2. 内部代号不外露。用户可见输出只保留必要严重度，例如 `S0 / S1 / S2 / S3`；不要输出 `P1 / PT1 / CC1 / ST1` 这类内部检查编号。
3. 每条问题都要讲影响。不要只说“28 列平铺”，要补一句“用户会拖横条找关键列，连续阅读被打断”。
4. 避免规范书腔调。少用“违反、偏离、承诺、兜底、硬塞、XX 级问题”等词，换成“发生了什么 + 会出什么事”。
5. 不堆加号并列。并列名词超过 3 个就拆句或分组，不用一串“原则 + 模式 + 契约 + 样式”压给用户。
6. 术语首次出现要解释。同一个概念在一份回答里只用一套叫法，例如 `Detail Drawer` 第一次写作“详情抽屉”，后面就只用“详情抽屉”。
7. Bullet 有节制。单块 bullet 最多 5 条；评审类输出要有开头结论和结尾收束，不要连续堆列表。

## UI 文案规则

- 已经有中文徽章、标签或状态时，不在同一视觉位置叠加原始英文枚举值。
- 按钮、标签、空状态、错误提示默认用中文；品牌名、代码字段、无通用译名的技术词保留原文。
- 提示和错误文案用“发生了什么 + 怎么办”，少用“校验不通过 / 发生异常 / 缺少权限”这种泛化说法。

## 自检

发送前读一遍：

1. 第一句是否已经给结论？
2. 是否暴露了内部编号？
3. 每个问题是否说明了用户或业务影响？
4. 是否有规范书腔调词？
5. 同一概念是否混用了多种叫法？
6. 列表之间是否有叙述承接？
